VPC Tips

VPC is a logical datacenter. Do not span regions, but can span availability zones. Can have a maximum of 5 VPCs in each AWS region by default Consists of Ineternet Gateways (IGWs) or Virtual Private Gateways, Route Tables, Access Control Lists, Subnets, Security Groups. 1 subnet = 1 availability zone (they cannot span multiple AZs) - this comes up a lot on exams It is a logically isolated section of AWS where you can AWS resources in a virtual network that you define. You have complete control over virtual networking env, including selection of your IP address range, creation of subnets, configuration of route tables and network gateways. Can create public facing subnet for webservers that has access to Internet, and place backend systems such as DBs or app servers in private facing subnet with no Internet access. Can leverage mutlple layers of security, including security group and network access control lists, to helpl control access to EC2 instances in each subnet. Can create Hardware VPN connection between your corporate datacenter and your VPC and leverage the AWS cloud as an extension of corporate datacenter. Subnets can be public or private. They will have different address ranges. Example: public 10.0.1.0/24, private 10.0.2.0/24. Security groups and Access Control Lists (ACLs) can span subnets. Can only have one internet gateway per VPC. Common question would be "in order to increase network speed, can you attach additional IGWs?". No, you cannot. Security groups are stateful. If you create a rule allowing HTTP into security group, by default HTTP is allowed out as well ACLs however are stateless. If you create a rule allowing HTTP in, you have to create the rule allowing HTTP out as well. Default VPC: Is user friendly, allowing you to immediately deploy instances. Is created by default when AWS account is created. All subnets in default VPC have a route out to the Internet. They are all public. Each EC2 instance within default VPC will have public and private IP. If deleted, the only way to get it back is to contact AWS. VPC Peering: Allows you to connect one VPC with another via a direct network route using private IP addresses (rather than going out through the Internet). Instances behave as if they're on the same private network Can peer VPCs with other AWS accounts as well as with other VPCs in the same account. Always done in a star configuration, i.e. 1 central VPC peers with 4 others. No transitive peering! No transitive peering means that if you have 5 VPCs (A-E), with B-E all peered to A, that means that B-E are not peered to each other even though they are all peered to A. You would need to peer them individually to each other if desired. Creating: When creating a VPC manually via the console, it will automatically create security groups, ACLs and route tables. It does not create a subnet or internet gateway. When creating a subnet, 3 IPs are reserved by default. Does not count .0 or .255 IPs. They are: .1 - VPC router .2 - DNS Server .3 - Reserved for future use .0 and .255 cannot be addressed, so they are not counted as reserved by AWS, but still cannot be used. When creating a new subnet, it will always be associated with your main route table. For that reason, you should not have a route out to the Internet on the main route table, because every subnet you create by default will have Internet access. NAT: NAT Instances: When creating you must disable source/destination check on the NAT instance. Must be in a public subnet Must be a route out of private subnet to the NAT instance in order for it to work Must have a public IP. Amount of traffic that instance supports depends on instance size. If it's bottlenecking, increase instance size. Can create high availability using autoscaling groups, multiple subnets in different AZs and a script to automate failover Always behind a security group Have to be patched to keep up with security updates NAT Gateways: New, may not be in exam yet Much easier to setup and maintain than NAT instances (though may cost a bit more) Preferred by the enterprise Scale up to 10Gbps automatically No need to patch Not associated with security groups Automatically assigned a public IP Need to update your route table to point to NAT Gateway No need to disable source/destination check Bastions: Used to securely administer EC2 instances (using SSH or RDP) in private subnets. Also called jump boxes or jump servers. SSH into bastion, and from there you can SSH into server in private subnet. Put them behind an autoscaling group with a minimum size of 2. Use Route53 to automatically fail over. NATs: Used to provide internet access to EC2 instances that are within private subnets. ACLs and Security Groups Security Groups: Security groups are first line of defense and are at the instance level Everything is denied by default, white list allows various protocols and IPs through Stateful, when opening up inbound traffic, outbound is automatically allowed as well Instances must be manually put into a security group ACLs: ACLs are second line of defense and are at the subnet level Allows both allow/deny rules so you can allow SSH, but block it from various IPs. Stateless, when opening up inbound traffic, outbound is not automatically allowed. Must be manually added. Automatically applied to all instances in the subnet. No need to add to each instance within the subnet. Rules are applied in numerical order. Lower numbered rules will trump higher numbered rules. When creating a VPC, an ACL is automatically created which allows all inbound/outbound traffic. But when creating an ACL, it will block all inbound/outbound traffic by default. ACLs can be associated with multiple subnets, however a subnet can only be associated with one ACL. If you associate an ACL with a subnet that already is associated to another ACL, the previous association is removed. If you don't associate a subnet with an ACL, it automatically gets associated with default ACL. To block specific IPs from a network, you can only use an ACL. Security groups do not have this option. Load Balancers: If you want your servers to be highly available, you always want at least 2 public subnets and 2 private subnets so that you can use a load balancer to send traffic to each. Make sure each subnet is in a different availability zone. When creating subnets within a VPC in different availability zones, by default they can still communicate with each other.
